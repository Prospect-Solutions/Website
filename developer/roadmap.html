Build strategy

Why this order

The wrong approach is to build by layer — infrastructure first, then agents, then UI. That produces an impressive-looking system that cannot be validated until everything is done. The right approach is to build by value: the narrowest possible end-to-end slice first, then widen.

→

The core principle

Every milestone ends with a system you can run, observe, and test. If a milestone fails to produce a working system, it is not complete. Partial layers do not count as milestones — working slices do.

What we're optimising for

Early validation over early completeness

A system that runs one agent end-to-end in week 4 teaches you more than a system that has five half-built agents in week 8. Each milestone produces evidence that the architecture works — before the next one is built on top of it.

Risk management

Each milestone is a natural stopping point

If the project pauses or changes direction after any milestone, the system at that point is usable. Nothing is half-built. M1 alone is a useful working tool. M2 alone is a usable business system. Each milestone stands on its own.

Agent contract first

The contract is built once in M1 and never changes

Every agent added in M2, M3, and M4 slots into the contract established in M1. This means adding agents never requires changes to the orchestration engine. The developer can build agents in parallel once the contract is locked.

Human loop from day one

Approval gates are live from M1 — never retrofitted

Building the HITL gate in M1 means every agent that follows is born with human oversight. Retrofitting approval gates into an already-running system is significantly harder and riskier than building them in from the start.

Framework before features

The first thing Oliver builds is not an agent — it is the plumbing that makes agents possible. API messaging layer, Claude API integration, agent contract schema, and the callback mechanism. Only once a message can travel from the orchestrator to an agent and back does it make sense to build the first real agent on top of it. Starting with the desktop app or UI is the wrong order.

→

Local machine → server decision

M1 runs entirely on Oliver's local machine. The server is not provisioned at the start — it is provisioned once real usage during M1 reveals the actual resource requirements. This avoids over- or under-speccing the server blind. The target is to have server requirements clear by the end of M1, with the server acquired at the M1/M2 boundary or during M2.

Server decision point

By the end of M1, Oliver will have real data on how long runs take, how much memory they use, and how often they need to run. That data determines the server spec. The server should be provisioned at the M1→M2 boundary — not before, and not after M2 starts. Railway or Fly.io are the default options; the decision can wait until the numbers are known.

Gate to M2 — do not start M2 until these pass

  • API messaging round-trip confirmed end-to-end with a synthetic payload
  • At least 3 full Curator runs completed and reviewed without errors
  • tools_and_services has ≥ 20 verified records across ≥ 4 categories
  • HITL gate tested with both approve and reject paths
  • Cost per run is logged and visible in the admin UI
  • Audit log correctly records every state change and approval decision
  • Agent swap tested: endpoint URL changed in registry, next run uses new endpoint
  • Server spec decision made — provisioning plan confirmed

Primary risk at this milestone

The agent contract is the highest-risk component in M1. If the input/output schema is underspecified, every subsequent agent will need changes when it hits an edge the contract did not anticipate. Spend extra time on the contract definition and test it with synthetic payloads before connecting a real agent. The framework must be solid before the first agent is built on top of it.

Client intake UI — simple and protected

The intake interface is a standalone chatbot widget that can be embedded on the prospect.solutions website via an iframe or script tag. It is not a polished client-facing product at this stage — it is a working tool for testing the pipeline. Access is gated by a simple access code. Rate limiting, max prompt counts, and message length limits prevent abuse. Session persistence (pause and resume) is deferred to M3.

Milestone 3

Automation — wire it up,
make it run itself

M3 is primarily wiring and automating what already exists — not building new things. The scheduler, queue, and event triggers connect the pieces built in M1 and M2 into a fully automated system. Straightforward because the building blocks are already in place. Server is required and confirmed here.

⏱ Weeks 10–14 · ~4–5 weeks · 1 developer · server confirmed
→

Mostly wiring, not building

Everything in M3 connects or automates something that already exists. The cron scheduler fires the Curator that already runs. The Redis queue manages the agents that already respond. The event triggers chain the pipelines that already work manually. Session persistence (deferred from M2) is added here. M3 is relatively low-risk because the components are proven — the challenge is integration sequencing and concurrency, not new functionality.

Milestone 4

generated, iterated, refined

The three final documents are generated automatically. But this milestone does not end when the generators are built — it ends when the output quality is genuinely good. That takes human review cycles, prompt iteration, and possibly extra help. This milestone may stretch.

⏱ Weeks 15–19 · ~4–6 weeks · 1–2 developers · heavy human review
!

Output quality will vary — plan for iteration

Generating a structurally correct document is the easy part. Generating a Project Proposal that is genuinely defensible, a Pitch Deck that communicates clearly, and a Custom Development Proposal that a client would take seriously — that takes iteration. The AI output will be variable, especially early in the milestone. Budget for multiple review cycles, real human judgment on each output, and the possibility of bringing in additional help to refine agent prompts. This milestone is not complete until the quality bar is met, not just the feature list.

The Oliver decision

M5 deliberately overlaps with the question of Oliver's ongoing engagement. By the time M5 begins, there will be real data: how much the system costs to run, how much developer time it needs, what the backlog of improvements looks like, and whether the business case for full-time development is clear. M5 is the right place to make that decision — not before, when it would be speculative.

→

After M5 — what comes next

Future work falls into three categories: (1) additional business branches — Accounting and Legal agents as defined in TM01; (2) database growth — expanding tools_and_services to cover more categories and geographies; (3) client pipeline improvements — prompt tuning on the Consultant and Analyst based on real engagement data. Each becomes a new Transition Moment in the document chain.
