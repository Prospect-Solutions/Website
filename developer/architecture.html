Architecture specification · Tier 3

A system that runs itself,
with humans in the loop

A fully automated, cloud-hosted business intelligence platform. Six AI agents — three workers, three validators — orchestrated by a model-agnostic engine. Agents are plug-in components: swap the model, change nothing else.

01 — Overview

Three design principles

Before any code. These three decisions shape every other choice in the system.

The system is independent of its agents

The scheduler, state machine, data layer, and approval gates exist without any specific AI model. Agents are workers hired to do a job — the factory floor does not change when you hire different workers. Swap Claude for GPT-4o or a fine-tuned model by changing one row in a database table. Nothing else changes.

Agents communicate through a contract, not a chat

Every agent — worker or validator — receives a standardised JSON input payload and must return a standardised JSON output payload. The orchestrator sends the job and waits for the result. It never knows or cares what happened inside the agent. This is what makes them plug-in components rather than hardwired pieces.

Human approval is a first-class system component

The human-in-the-loop gate is not an afterthought or a manual step you do separately. It is a named state in the run lifecycle with its own timeout, escalation path, and notification logic. If no approval arrives within the configured window, the system escalates and alerts — it does not block silently.

→

The key shift from Tier 1

On Tier 1, you were the orchestrator — you opened chats, triggered agents in sequence, and moved data manually. On Tier 3, the orchestration engine does all of that automatically. You only appear at the human-in-the-loop gate, where your judgment is genuinely needed.

02 — System layers

Five layers, one system

Each layer has exactly one job. Data flows downward — never upward. Agents never write directly to the presentation layer. This section specifies every component inside each layer in enough detail for a developer to build it.

!

How to read this section

Each layer is broken into its individual components. For each component you will find: what it does, its internal logic or data structure, its interface with adjacent components, and implementation notes. This is the developer reference — not a summary.

Component 4 — Agent router

Looks up the registered endpoint for an agent type, assembles the input payload, POSTs it, and handles the async callback. The router is what makes agents swappable — it reads from the agents registry table at dispatch time, so a model swap takes effect immediately on the next run without a code deploy.

Dispatch logic
async function dispatch(job) { // 1. Look up the registered agent for this job type const agent = await db.agents.findActive(job.agent_type); if (!agent) throw new Error(`No active agent registered for type: ${job.agent_type}`); // 2. Assemble standardised input payload const payload = { job_id: job.id, agent_type: job.agent_type, config: await config.getForAgent(job.agent_type), context: await context.buildFor(job), callback_url: `${process.env.BASE_URL}/api/callbacks/${job.id}`, timeout_seconds: agent.timeout_seconds }; // 3. POST to agent endpoint with auth header const res = await fetch(agent.endpoint_url, { method: 'POST', headers: { 'Content-Type': 'application/json', 'X-Agent-Secret': agent.secret_hash }, body: JSON.stringify(payload), signal: AbortSignal.timeout(agent.timeout_seconds * 1000) }); if (!res.ok) await jobs.handleDispatchError(job.id, res.status); } // Callback handler — agent POSTs result here when done async function handleCallback(jobId, result) { // Validate result schema before accepting const valid = schema.validate(result, agentOutputSchema); if (!valid) return jobs.handleInvalidResult(jobId, result); await db.jobs.storeResult(jobId, result); await jobs.transition(jobId, 'validating'); }

Layer 3

Agent layer

The microservice wrapper every agent must implement — independent of which AI model is inside

→

This section specifies the wrapper, not the prompts

Every agent is a small Node.js (or Python) microservice that wraps an AI model call. This section specifies that wrapper — the HTTP server, config loading, the AI call structure, error handling, and how the result is packaged. The AI prompts for each specific agent are specified in Section 03.

Agent microservice structure

Every agent — all six — is a separate deployable service with this exact structure. The only parts that differ between agents are the system prompt, the context builder, and the result formatter.

// Minimal agent server — same structure for all 6 agents const express = require('express'); const app = express(); app.use(express.json({ limit: '2mb' })); // Single endpoint — receives job from orchestrator app.post('/run', validateSecret, async (req, res) => { const { job_id, config, context, callback_url, timeout_seconds } = req.body; // Acknowledge immediately — agent runs async res.json({ status: 'accepted', job_id }); try { const result = await runWithTimeout( () => agentLogic(config, context), timeout_seconds * 1000 ); await postCallback(callback_url, { job_id, status: 'success', ...result }); } catch (err) { await postCallback(callback_url, { job_id, status: 'failed', errors: [{ code: err.code || 'AGENT_ERROR', message: err.message }], run_log: err.partialLog || '' }); } }); // Health check — orchestrator polls this to verify agent is alive app.get('/health', (req, res) => res.json({ status: 'ok', agent_type: process.env.AGENT_TYPE, version: process.env.VERSION })); app.listen(3000);

Adding a new external service

Any new third-party API is added at Layer 5 only. The agent that needs it gets the new env variable in its service config. No other layer changes. This isolation is what prevents a service swap (e.g. moving from Brave Search to Serper) from requiring system-wide changes.

Internal logic — 9 steps in order
1
Load config — read all parameters from the config object. Set operating limits for this run.
2
Mode 3 — stale refresh — for each record in context.stale_records: re-fetch homepage and pricing page. Compare to stored values. If unchanged: update last_updated only (write directly to live sheet via Sheets API). If changed: create a draft row with change summary in curator_notes.
3
Mode 1 — web search — for each category in config.categories: run 3–4 search queries. Evaluate each result against the quality checklist. Build draft rows for novel candidates. Stop when max_new_drafts_per_run is reached.
4
Mode 2 — source monitoring — visit each URL in config.monitored_sources. Extract newly listed tools. Cross-check against existing_ids and existing_domains. Build draft rows for novel candidates.
5
Deduplication pass — scan the current draft batch for internal duplicates (same id or same domain). Merge or remove. Every draft must be unique.
6
Confidence scoring — for each draft: run the 8-signal confidence check. Calculate confidence_score and source_count. Auto-reject any draft below config.confidence_score_minimum.
7
Structural validation — run all 12 pre-submission checks on every draft row. Remove failing rows and log reasons.
8
Write to tools_draft — write all passing rows to the tools_draft Google Sheet via Sheets API. Each row includes all 17 base fields + source_urls + discovery_source + curator_notes.

Worker · Pipeline 2

Consultant

Conducts structured client intake conversations and produces Problem Memos

TriggerClient session start event
Runtime30–120 min (client-paced)
ConcurrencyUp to 5 parallel sessions
ModelClaude Sonnet (configurable)
How this agent differs from the others

The Consultant is the only agent that runs an interactive, multi-turn conversation rather than a single AI call. It does not complete in one shot — it exchanges messages with a human client over minutes or hours. The session state is persisted in Postgres between every message so the conversation survives browser refreshes, network drops, and the client walking away and coming back.

Input context object
context.session_iduuidThe client session this conversation belongs to
context.client_namestringClient's name — used to personalise the conversation
context.client_companystringClient's company name
context.conversation_historyobject[]All previous messages in this session [{role, content}]. Sent on every turn so the agent has full context.
context.required_fieldsstring[]The Problem Memo fields that must be populated before the memo can be finalised — from agent_config
context.new_messagestringThe client's latest message — what the agent is responding to on this turn
Internal logic — conversation state machine
1
Determine conversation phase — based on conversation_history, determine which required_fields are still unpopulated. This determines what to ask next.
2
Generate response — make AI call with full conversation_history as context. The agent asks at most one question per turn. It uses the client's own language, not technical terms.
3
Extract field values — after each client message, run a structured extraction pass to identify which required_fields are now answered. Update the running memo draft.
4
Check completeness — if all required_fields are populated AND the problem is unambiguous: transition to confirmation phase.
5
Confirmation phase — present a plain-English summary of the Problem Memo to the client. Ask them to confirm or correct. Do not proceed until they confirm.
6
Finalise memo — write the completed Problem Memo to the problem_memos Postgres table. Emit memo_created event. Return result to orchestrator.

Worker · Pipeline 3

Analyst

Matches Problem Memos to the tools database and produces ranked Solution Presentations

Triggerintake_validation_passed event
Runtime3–10 min
ConcurrencyUp to 3 parallel
ModelClaude Sonnet (configurable)
Input context object
context.memoobjectThe full Problem Memo object — all fields from the problem_memos table
context.tools_snapshotobject[]All active rows from tools_and_services — the full database the Analyst queries. Passed in as context rather than read directly, so the agent gets a consistent snapshot.
context.max_resultsintegerMaximum number of tools to include in the presentation (from config, default 5)
context.min_confidencefloatMinimum tool confidence_score to consider for matching (from config, default 0.4)
Internal logic — objective-based matching algorithm

The Analyst scores solutions by the proportion of the memo's objectives they meet — not by generic tool quality. A solution can be a single tool or a combination of compatible tools. If no solution clears the 40% objectives threshold, a custom development proposal is generated instead.

1
Pre-filter — remove all tools where confidence_score < context.min_confidence or status ≠ active. This reduces the candidate set before the expensive matching step.
2
Objective mapping — for each tool in the candidate set, assess which objectives from memo.objectives it addresses. Record as a boolean map: {objective_id: true/false}. Objective coverage % = objectives met ÷ total objectives.
3
Single-tool pass — rank tools by objective coverage %. Keep all tools clearing a minimum threshold (configurable, default 20%). Discard the rest.
4
Combination pass — for tools that individually cover 20–60% of objectives: test pairings and triplets. Check integration compatibility before combining (verify both tools appear in each other's integrations field, or have a documented connector). Score the combination by the union of their covered objectives.
5
Constraint filtering — remove solutions (single or combined) that violate hard constraints: budget, company size fit, tech_stack incompatibility, missing required integrations from memo.
6
40% threshold check — does any remaining solution (single or combination) meet ≥ 40% of the memo's objectives? If yes: continue to step 7. If no: route to custom development proposal path (step 9).
7
Final ranking — rank passing solutions by: (objective_coverage % × 0.70) + (confidence_score × 0.20) + (constraint_fit × 0.10). Take top context.max_results solutions.
8
Generate rationale — for each top solution: produce a rationale that maps which specific objectives it meets, which assumptions it challenges, and why it fits the client's tech_stack. Assemble the Solution Presentation and Project Proposal. Emit solution_created event.
9
Custom development proposal path — if no solution clears 40%: generate a Custom Development Proposal. This document states that a custom-built system is preferable, justifies why based on the search outcome and problem memo (citing which objectives no existing tool covers), and outlines a preliminary project scope. Proposal is assembled as a structured document and saved to Drive. Emit custom_proposal_created event.

Validator · Pipeline 1

DB Validator

Independently verifies each draft tool record before it reaches the admin review queue

Triggercurator_complete event
Runtime15–30 min
Concurrency1 (serial — one per curator run)
ModelClaude Sonnet + web search
What it validates and why

The DB Validator is the quality gate between the Curator's output and the admin's review queue. It catches two failure modes the structural validation cannot catch: (1) problem_tags or embedding_text that are technically valid but semantically wrong — e.g. tags written in feature language rather than client language, or an embedding_text that misrepresents the tool. (2) Tools that the Curator found but could not be independently corroborated — possible hallucinations or defunct tools.

Input context object
context.draft_recordsobject[]All rows currently in tools_draft with empty quality_score — the records to validate
context.quality_thresholdfloatScore below which validator_flag is set to TRUE (from config, default 3.0)
Scoring rubric — applied to every draft record
DimensionWhat it measures5 — Excellent3 — Acceptable1 — Fail
problem_tags accuracy Do the tags describe real problems this tool solves, in client language? All tags accurate, specific, written as a client would say them Broadly correct but some tags are vague or slightly technical Tags are wrong, copied from marketing, or feature descriptions
embedding_text accuracy Does the description accurately represent what the tool does? Fully accurate, clear, objective, complete — confirmed by independent sources Broadly correct but missing a key differentiator or audience Significantly wrong, misleading, or copied from the tool's own marketing
independent confidence How well-established and verifiable is this tool from sources the Curator did not use? Well-known, many independent reviews, clearly active, all details verifiable Smaller tool, fewer reviews, but clearly real and findable Almost no independent presence — cannot verify key claims

quality_score = average of three dimensions, rounded to 1 decimal. Records below quality_threshold have validator_flag set to TRUE.

Per-record logic
1
Web search — search for the tool by name using at least 2 independent sources (not the tool's own website). Store the URLs found.
2
Score all three dimensions — make one AI call per record with the draft fields + independent source content. Ask for scores with reasoning.
3
Generate correction suggestions — if any dimension scores below 3: produce a corrected version prefixed with "SUGGESTED CORRECTION:" in quality_notes. Never overwrite original fields.
4
Write scores to tools_draft — update the record's quality score columns. Set validator_flag = (quality_score < threshold).
Output — columns written to tools_draft
quality_scorefloatAverage of three dimension scores (1.0–5.0)
quality_tags_scoreintegerproblem_tags accuracy dimension (1–5)
quality_embedding_scoreintegerembedding_text accuracy dimension (1–5)
quality_confidence_scoreintegerIndependent corroboration dimension (1–5)
quality_notesstringReasoning for scores + any suggested corrections
quality_sourcesstringComma-separated URLs the Validator used — separate from the Curator's source_urls
validator_flagbooleanTRUE if quality_score < quality_threshold — surfaced prominently in admin review UI

04 — Orchestration

Every run follows the same lifecycle

Whether it's the Curator doing a weekly database scan or the Analyst matching a client memo — every job goes through the same seven states, managed by the orchestration engine.

1
state: queued

Trigger fires — job record created

A cron schedule, an incoming client event, or a manual trigger creates a job record in Postgres. Status is set to queued. Config is loaded from the agent_config table. The job is pushed onto the Redis queue.

2
state: running

Router dispatches the worker agent

The agent router reads the job type, looks up the registered endpoint for that agent in the agents table, and POSTs the standardised input payload. Status → running. A timeout clock starts.

3
state: running

Validator agent dispatched in sequence

When the worker posts its result back, the router immediately dispatches the corresponding validator agent with the worker's output as its input context. Both worker and validator results are stored in the job record.

4
state: validating → pending_review

Results packaged and surfaced for review

The orchestrator assembles the full results — worker output, validator scores, flagged items — into a review payload. Status → pending_review. Notification sent to the configured admin email.

5
state: pending_review

Human-in-the-loop gate — approval required

The admin reviews results in the dashboard. Each item can be approved, edited, or rejected. A timeout clock runs: if no action is taken within the configured window (default 72 hours), an escalation alert fires. The system waits — it does not auto-approve or auto-reject.

6
state: committing

Approved results committed to the data layer

For each approved item: the orchestrator writes it to the appropriate destination (tools_and_services, a client-facing document, etc.), exports any required artifacts, and records the approval decision with timestamp and approver identity.

7
state: completed

Job closed — audit record finalised

Status → completed. Duration, item counts, approval rates, and any errors are written to the audit log. The job record is sealed and immutable.

Why the contract matters

Without this contract, every model change requires system changes. With it, the model is just a detail of the agent's internal implementation — the system is completely decoupled from it. This is also why agents can be developed, tested, and deployed independently, in parallel, by different developers.

06 — Data layer

State lives in the data layer, never in agents

Agents are stateless. Between runs, they hold nothing. All state — system state, business data, files — lives in the data layer. This is what makes the system restartable, auditable, and recoverable from failure.

PG

Postgres — system of record

The authoritative store for everything the system owns. Job records, run history, approval decisions, the agent registry, config. Every state change is a write to Postgres first.

→ jobs (run lifecycle) → agents (registry + endpoints) → approvals (decision + actor) → audit_log (immutable) → agent_config (parameters)
GS

Google Sheets — business data

The human-readable, agent-queryable store for business records. tools_and_services is the database the Analyst queries. document_register is the metadata index over Drive files.

→ tools_and_services → tools_draft (staging) → document_register → agent_config (parameters)
GD

Google Drive — file storage

The file system. Client memos, solution presentations, contracts, invoices, reports. Files are referenced by Drive ID in the document_register — never accessed by path.

→ /Operational/Problem Memos/ → /Operational/Solution Presentations/ → /Legal/ → /Accounting/ → /Organization/Architecture/
RQ

Redis + BullMQ — job queue

The async job queue between the scheduler and the agent runner. Handles priority (urgent client intake over routine curator run), deduplication (no two identical jobs at once), and retry with exponential backoff on failure.

→ priority queues per agent type → retry with backoff (max 3) → dead letter queue for failures → job deduplication by type+date
!

The upgrade path is already built in

Google Sheets is the right starting point — it's already set up, free, and human-readable. But the schema field types are Postgres-compatible by design. When tools_and_services grows beyond ~5,000 rows or you need complex relational queries, migrating to Supabase (managed Postgres) requires a data export and schema creation — not a system redesign.

07 — Human in the loop

Automation with oversight built in

The human approval gate is not a manual workaround — it is a designed component with state, timeouts, and escalation. You appear exactly where your judgment is needed, and nowhere else.

Step 1

System notifies you

Email (or Slack) with a summary: how many items, validator flags, confidence scores. Link directly to the review dashboard.

Step 2

You review in the dashboard

Each item shows: worker output, validator score, quality notes, suggested corrections, source URLs. You decide: approve, edit, or reject.

Step 3

System commits your decisions

Approved items are written to the live data layer with your name and timestamp attached. Rejected items are logged with reason. Exports fire automatically.

What happens if you don't review in time

AT 48 HOURS

First escalation alert

Second notification sent. The draft batch is still waiting. The next scheduled run will be blocked until this one is cleared.

AT 72 HOURS (CONFIGURABLE)

Deadline alert

Final alert. The job is marked expired in the audit log. The draft is preserved but the next run is unblocked.

→

The system never auto-approves

No agent output ever reaches the live database without a human approval decision. This is a hard architectural constraint, not a configurable setting. The only exception is stale record refreshes where the content did not change — those write a date update directly, since no judgment is required.
